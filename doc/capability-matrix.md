## Rules

- A capability does not become stable merely because code exists.
- Any new stable `Harness` API requires a matrix update before or with the implementation.
- Any new test-only mock must document seed state, capture behavior, failure injection, reset
  semantics, and a minimal usage example.
- Any feature that depends on HTML or DOM behavior must be checked against `../html-standard/`
  first.
- The capability matrix is the source of truth for what the Go workspace promises.
- Legacy and deprecated spec branches are not default targets; only current, bounded, user-visible
  behavior should be added unless the matrix explicitly calls out a compatibility exception.
